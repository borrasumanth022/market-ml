## Why OOS Deltas Are Not Reliable Here

The OOS walk-forward F1 deltas in this step are **structurally inflated** and cannot
be used as the decision signal.

**Root cause:** The shared sector model was trained on *all* pre-holdout data. When
fine-tuning fold k, the warm-start base already contains trees fitted on the fold's
validation rows. The +100 additional estimators then reinforce those patterns further.
Every validation fold is effectively an in-sample evaluation of the base model component,
producing artificially high F1 scores (+0.09 to +0.15 above shared model baseline).

**The only valid signal is holdout F1** (>= 2024-01-01, never seen during any training).

**Correct approach for future fine-tuning:** Hold out a dedicated calibration/validation
split (e.g. 2022-01-01 to 2023-12-31) from pre-holdout data. Train the base shared model
on data before 2022. Fine-tune on the 2022-2023 window. Evaluate on the 2024+ holdout.
This eliminates warm-start leakage from the walk-forward evaluation entirely.

## Lesson Learned: Walk-Forward on a Pre-Trained Base

Standard walk-forward evaluation assumes the model is trained fresh on each fold's
training data. When warm-starting from a pre-trained base model that was fitted on
the *entire* pre-holdout dataset, the standard validation guarantee breaks down:

- Each fold's validation set was already seen by the base model
- The +100 trees exploit those memorised patterns
- F1 on validation folds looks like improvement but is partly base-model recall

**Do not use OOS walk-forward deltas as the save criterion when warm-starting.**
Use a clean held-out split that the base model has never touched.
